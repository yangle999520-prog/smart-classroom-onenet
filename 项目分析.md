# 智能教室环境监测系统：项目分析

## 1. 项目概览

本项目是一个基于 STM32、ESP8266 和 OneNET 的智能教室环境监测系统，采集教室温度、光照等数据，控制 LED 照明，并提供 Web 可视化监控页面。

项目当前包含四类内容：

| 模块 | 目录 | 主要职责 | 当前状态 |
| --- | --- | --- | --- |
| 后端 | `backend` | 接入 OneNET、保存传感器数据、提供 REST/WebSocket 接口 | 主业务模块 |
| Web 前端 | `frontend` | 实时仪表盘、历史曲线、统计图表 | 主展示模块 |
| 嵌入式端 | `stm32+esp8266` | 采集传感器、自动控制 LED、通过 MQTT 上报 | 需要 Keil/硬件 |
| uni-app | `微信小程序4.0` | 移动端设备监控与控制 | 已实现主要页面和接口调用 |

## 2. 总体架构

```text
DS18B20 温度传感器       光照传感器
          \                  /
           STM32F103 + ESP8266
                    |
          MQTT / OneNET 上报与下发
                    |
          OneNET Pulsar 服务端订阅
                    |
        Spring Boot 后端（8080）
          |                  |
       MySQL             WebSocket
          |                  |
          +---- Vue3/Vite 前端（3000）
```

主要数据流如下：

1. STM32 周期读取 DS18B20 和光照传感器。
2. 光照低于 30% 时，自动模式下打开 LED；否则关闭 LED。
3. ESP8266 连接 `studio-mqtt.heclouds.com:1883`，通过 OneNET MQTT 上报数据。
4. 后端 `OneNetPulsarSubscriber` 订阅 OneNET Pulsar Topic，解析多种消息格式。
5. 后端将完整数据写入 MySQL `sensor_data` 表，并广播到 WebSocket 客户端。
6. Vue 前端通过 REST 获取初始数据，通过 `/ws/sensor` 接收实时数据。

## 4. 后端分析

### 4.1 技术栈

- Java 8
- Spring Boot 2.7.18
- Spring Web、Spring Data JPA、WebSocket
- MySQL Connector/J
- Apache Pulsar Client 3.0.1
- FastJSON 2.0.32
- Eclipse Paho MQTT（代码中保留，但当前配置未启用 MQTT 客户端）

### 4.2 数据模型

实体 `SensorData` 对应表 `sensor_data`：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `id` | BIGINT | 自增主键 |
| `temperature` | FLOAT | 温度，单位 °C |
| `light` | INT | 光照百分比 |
| `ledStatus` | TINYINT | LED 状态，0 关、1 开 |
| `mode` | TINYINT | 工作模式，0 自动、1 手动 |
| `createTime` | DATETIME | 数据采集/入库时间 |

历史数据默认查询最近 24 小时，后端将小时范围限制在 1～720 小时。统计数据基于最近 24 小时计算。

### 4.3 REST 接口

所有接口前缀为 `/api/sensor`，成功响应通常为 `{ code: 200, data: ... }`。

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/upload` | 手动或模拟上传传感器数据 |
| GET | `/latest` | 查询最新一条数据 |
| GET | `/history?hours=24` | 查询指定小时内的历史数据 |
| GET | `/list?page=0&size=20` | 分页查询全部数据 |
| GET | `/statistics` | 查询近 24 小时统计信息 |
| GET | `/device-status` | 调用 OneNET API 查询设备在线状态 |

上传示例：

```json
{
  "temperature": 25.6,
  "light": 42,
  "ledStatus": 0,
  "mode": 0
}
```

### 4.4 实时通信

- WebSocket 地址：`ws://localhost:8080/ws/sensor`
- 前端开发环境地址：`ws://localhost:3000/ws/sensor`，由 Vite 代理到后端。
- 服务端只负责广播，不处理前端发来的控制消息。
- Pulsar 消费到完整数据后才入库并广播，至少需要温度和光照字段。

### 4.5 前端功能

`Dashboard.vue` 提供：

- 温度、光照、LED、工作模式的最新状态展示；
- 温度和光照状态阈值提示；
- 最近数据列表；
- OneNET 设备在线状态；
- WebSocket 断线后每 3 秒自动重连。

`History.vue` 提供：

- 1、6、12、24 小时以及 7、30 天时间范围切换；
- 温度趋势图；
- 光照趋势图；
- LED 开关分布饼图；
- 温度-光照散点图。

### 4.6 小程序功能

小程序主要代码位于 `微信小程序4.0/pages/index/index.vue`，当前已形成可用的移动端设备控制面板，主要包括：

- 显示设备在线/离线状态；
- 每 3 秒通过后端 `/api/sensor/latest` 拉取最新温度、光照、LED 和工作模式；
- 每 10 秒通过后端 `/api/sensor/device-status` 刷新 OneNET 设备状态；
- 自动/手动工作模式切换；
- 手动模式下控制台灯开关；
- 温度计式进度显示、光照进度条和 LED 状态卡片；
- 页面隐藏或销毁时清理定时器，避免重复请求；
- 操作后设置短暂状态锁，避免后端旧数据立即覆盖用户刚刚发起的操作。

小程序查询数据和设备状态经过 Spring Boot 后端，控制指令则直接调用 OneNET `thingmodel/set-device-property` 接口。`微信小程序4.0/key.js` 中配置了后端局域网地址和 OneNET Token 生成逻辑。

## 5. 嵌入式端分析

STM32 主循环每 10 ms 运行一次：

- 每约 100 ms 采样一次传感器；
- 每约 400 ms 上报一次数据；
- 每约 1 s 输出一次串口日志；
- 自动模式下按光照 30% 阈值控制 LED；
- 接收并解析 OneNET 下发消息，可处理 LED 和模式相关控制。

固件工程为 Keil 工程，已有 `stm32+esp8266/output/stm32f103.hex` 可供烧录参考。实际烧录前仍需确认芯片型号、串口波特率、Wi-Fi、OneNET 产品/设备参数和 GPIO 接线。

## 6. 优点与当前问题

### 优点

- 前后端分层清晰，REST 与 WebSocket 职责明确。
- OneNET 数据接入已从轮询升级为 Pulsar 服务端订阅，实时性更好。
- 后端支持多种 OneNET 消息结构，具备一定兼容性。
- 前端提供实时监控、历史分析和设备在线状态，功能闭环较完整。
- 数据库查询按时间建立索引，满足当前历史查询场景。

### 风险与改进建议

1. `backend/src/main/resources/application.yml` 中包含 OneNET 密钥和数据库密码，建议立即更换已暴露凭据，并改用环境变量或外部配置。
2. `schema.sql` 使用 `DROP DATABASE IF EXISTS`，不适合生产环境启动或重复执行，建议拆分为初始化脚本和迁移脚本。
3. `onenet.pulsar.enabled` 当前为 `true`，无 OneNET 订阅条件时后端启动会持续连接失败；本地开发可先设为 `false`，用 `/upload` 验证链路。
4. CORS 和 WebSocket 当前允许任意来源，生产环境应限制为实际前端域名，并增加认证。
5. `/upload` 使用 `Map<String,Object>` 且缺少范围校验，建议改为 DTO + Bean Validation，校验温度、光照、状态值和模式值。
6. WebSocket 会话保存在单机内存中，集群部署时需要消息代理或统一推送层。
7. 数据表只有 `create_time` 索引，数据量增大后应增加归档策略、分区或按设备建模。
8. 前端部分文件存在中文编码异常，建议统一保存为 UTF-8，并检查构建产物中的界面文案。
9. 当前数据模型未包含设备 ID，多设备扩展时需要增加 `productId/deviceName/deviceId` 等维度。
10. 小程序 `key.js` 中包含后端局域网地址和 OneNET AccessKey，建议改为构建环境配置，并避免在客户端长期暴露高权限密钥。
11. 小程序目前通过定时轮询获取最新数据，后续可考虑 WebSocket 或订阅式推送以降低延迟和请求量。

## 7. 推荐演进路线

1. 先修复密钥管理、编码和数据库脚本的安全/可维护性问题。
2. 增加开发 profile：关闭 Pulsar，启用数据模拟器或提供稳定的测试数据脚本。
3. 为上传接口、OneNET 消息解析和统计服务补充单元测试。
4. 增加设备维度、用户权限、告警记录和控制指令审计。
5. 为前端增加 WebSocket 状态降级策略：断线时定时调用 `/latest`，避免页面完全失去更新。
6. 在现有小程序控制面板基础上补充错误提示、登录鉴权、操作结果回显和配置分离。
